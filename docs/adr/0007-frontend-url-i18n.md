# 7. Frontend i18n by localized URL

## Status

Accepted

## Context

`frontend` is public and indexed. Search engines must see one canonical page
per route, and visitors must land in their own language. `admin` (authenticated,
not indexed) already uses paraglide with a single `en` locale and no URL
localization; `frontend` needs more: locale-prefixed, translatable URLs, a
locale redirect, and a canonical link. Only `en` ships today, but adding `fr`
must not require re-plumbing.

## Decision

Localize by URL with paraglide's `url` strategy on TanStack Start (SSR).

- Every route carries a locale prefix (`/en/...`); path segments are
  translatable per locale. `urlPatterns` are generated by a typed helper
  (`src/i18n/translated-pathnames.ts`, adapted from TanStack's
  `start-i18n-paraglide` example) keyed off `routeTree.gen`, so a new route with
  no translation entry fails to compile. Adding `fr` is one locale column per
  entry plus `messages/fr.json`.
- `strategy: ["url", "cookie", "preferredLanguage", "baseLocale"]`. Bare `/` has
  no URL locale, so `paraglideMiddleware` (in `src/server.ts`) resolves it via
  cookie, then `Accept-Language`, then falls back to `en`, and redirects to the
  prefixed URL.
- The router `rewrite` (`input: deLocalizeUrl`, `output: localizeUrl`) keeps the
  route tree locale-agnostic. The server entry passes the original `req` to the
  middleware, not the callback's `request`, to avoid a double-delocalize
  redirect loop.
- `__root.tsx` sets `<html lang>` from `getLocale()` and emits
  `<link rel="canonical">` pointing at the `en` variant of the current path via
  `localizeHref(path, { locale: "en" })`.

## Consequences

- One canonical (English) URL per route for search engines; visitors are
  redirected to their language from `/`.
- Shipping `fr` is a data change (a locale column + a message file), not a
  rearchitecture.
- Extends ADR 0005: `frontend` reads the shared design-system message file via
  the array `pathPattern`.
